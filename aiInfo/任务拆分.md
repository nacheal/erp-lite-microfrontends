## 1. 文档说明

### 1.1 目的

本文档定义 ERP Lite 微前端系统的整体架构，包括应用拆分策略、技术选型、通信机制、样式隔离方案以及部署结构。作为项目开发的顶层指导文件，所有子应用的开发与集成均应遵循本文档规范。

### 1.2 适用范围

本文档适用于参与 ERP Lite 项目的所有开发人员，涵盖主应用（Shell）及四个子应用（用户管理、商品管理、订单管理、数据看板）的开发、集成与部署工作。

---

## 2. 背景与目标

### 2.1 项目背景

ERP Lite 是一套面向中型企业的内部管理平台，包含用户管理、商品管理、订单管理、数据看板等核心业务模块。各模块由不同团队独立维护，存在以下核心诉求：

- 模块间技术栈差异（部分团队使用 Vue，部分使用 React）
- 各模块需要独立发布，互不影响
- 需要统一的 UI 壳层（导航、权限、主题）
- 希望降低单体应用带来的构建时间过长、部署风险高等问题

### 2.2 架构目标

- 实现各子应用技术栈无关、独立开发、独立部署
- 主应用提供统一的用户认证、权限控制和 UI 框架
- 子应用间支持安全的跨应用通信
- JS 沙箱隔离，防止全局变量污染
- CSS 样式隔离，防止各应用间样式冲突
- 公共依赖共享，避免重复加载 React / Vue 运行时

### 2.3 非目标

- 本期不做 SSR（服务端渲染）支持
- 本期不做移动端适配
- 不要求 100% 的 IE 兼容性

## 3. 整体架构

### 3.1 架构模式

采用「主应用 + 多子应用」的微前端架构模式，主应用作为容器（Shell），负责路由调度和子应用生命周期管理；各子应用作为独立的 SPA，自包含完整的业务逻辑。

```
┌─────────────────────── 浏览器 ────────────────────────────┐
│                                                            │
│   ┌──────────────────── Shell App ───────────────────┐    │
│   │  顶部导航 | 侧边菜单 | 登录鉴权 | 全局状态        │    │
│   │                                                   │    │
│   │  ┌─────────────── 子应用挂载区 ───────────────┐  │    │
│   │  │                                             │  │    │
│   │  │  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  │  │    │
│   │  │  │ 用户 │  │ 商品 │  │ 订单 │  │ 数据 │  │  │    │
│   │  │  │ 管理 │  │ 管理 │  │ 管理 │  │ 看板 │  │  │    │
│   │  │  │React │  │Vue 3 │  │React │  │原生JS│  │  │    │
│   │  │  └──────┘  └──────┘  └──────┘  └──────┘  │  │    │
│   │  └─────────────────────────────────────────────┘  │    │
│   └───────────────────────────────────────────────────┘    │
└────────────────────────────────────────────────────────────┘
```

| 应用名称 | 类型 | 技术栈 | 本地端口 | 激活路由前缀 | 负责团队 |

## 4. 技术选型

### 4.1 微前端框架

| 方案 | JS 沙箱 | CSS 隔离 | 接入成本 | 社区成熟度 | 结论 |
|------|---------|----------|----------|------------|------|
| qiankun | ✅ 快照 + Proxy | ✅ Shadow DOM / scoped | 低 | 高 | ✅ 选用 |
| single-spa | 需自行实现 | 需自行实现 | 高 | 高 | ❌ 排除 |
| Module Federation | 弱 | 弱 | 中 | 中 | ❌ 排除 |
| iframe | 强（天然隔离） | 强（天然隔离） | 低 | — | ❌ 通信差 |

**选用 qiankun 的核心理由**：开箱即用的 JS 沙箱（Proxy 沙箱）与 CSS 隔离，接入成本低，社区文档完善，是目前企业级微前端落地的主流方案。

### 4.2 各应用技术栈

| 层级 | 技术/库 | 版本 | 说明 |
|------|---------|------|------|
| 构建工具 | Webpack 5 | 5.x | 主应用 & React 子应用 |
| 构建工具 | Vite | 5.x | Vue 子应用 & Dashboard |
| 主应用框架 | React | 18.x | Shell App |
| 子应用框架 | React | 18.x | 用户管理、订单管理 |
| 子应用框架 | Vue | 3.x | 商品管理 |
| 路由 | React Router | 6.x | React 应用 |
| 路由 | Vue Router | 4.x | Vue 应用 |
| 状态管理 | Zustand | 4.x | React 应用轻量状态 |
| UI 组件库 | Ant Design | 5.x | React 应用 |
| UI 组件库 | Element Plus | 2.x | Vue 应用 |
| HTTP 客户端 | Axios | 1.x | 所有应用 |
| 微前端框架 | qiankun | 2.x | 主应用集成 |

## 5. 路由设计

### 5.1 路由规划

路由采用「主应用控制一级路由 + 子应用内部管理二级路由」的策略。主应用根据 URL 前缀决定激活哪个子应用，子应用内部路由完全自治。

| URL 路径 | 归属 | 说明 |
|----------|------|------|
| /login | 主应用 | 登录页，未登录时重定向到此 |
| / | 主应用 | 首页，重定向到 /dashboard |
| /user/* | app-user 子应用 | 用户管理模块所有路由 |
| /user/list | app-user | 用户列表 |
| /user/detail/:id | app-user | 用户详情 |
| /user/role | app-user | 角色管理 |
| /product/* | app-product 子应用 | 商品管理模块所有路由 |
| /product/list | app-product | 商品列表 |
| /product/edit/:id | app-product | 商品编辑 |
| /order/* | app-order 子应用 | 订单管理模块所有路由 |
| /order/list | app-order | 订单列表 |
| /order/detail/:id | app-order | 订单详情 |
| /dashboard/* | app-dashboard 子应用 | 数据看板模块所有路由 |

## 6. 应用通信机制

### 6.1 通信方式总览

| 场景 | 通信方向 | 推荐方案 | 说明 |
|------|----------|----------|------|
| 传递用户信息 | 主 → 子 | qiankun props | 登录后 token / userInfo 下发 |
| 传递全局配置 | 主 → 子 | qiankun props | 主题色、语言、权限列表 |
| 子应用触发跳转 | 子 → 主 | qiankun initGlobalState | 子应用内需要跳转到另一子应用 |
| 跨子应用事件 | 子 → 子 | 全局事件总线 | 如订单模块跳转到用户详情 |
| 子应用通知主应用 | 子 → 主 | qiankun initGlobalState | 例如登出操作 |
| 子应用内部状态 | 子应用内部 | Zustand / Pinia | 不跨应用，各自独立管理 |

### 6.2 全局状态数据结构

主应用通过 qiankun 的 `initGlobalState` 维护以下全局状态，所有子应用只读，主应用可写：

```typescript
interface GlobalState {
  user: {
    id: string;           // 用户 ID
    name: string;         // 用户名
    token: string;        // JWT Token
    permissions: string[]; // 权限码列表
  } | null;
  theme: 'light' | 'dark';    // 主题模式
  locale: 'zh-CN' | 'en-US'; // 语言
  action?: {                  // 跨应用指令
    type: string;             // 指令类型
    payload: unknown;         // 指令参数
  };
}
```

### 6.3 跨子应用通信示例

**场景：订单模块点击用户名，跳转到用户管理并高亮该用户**

1. 订单子应用调用 `setGlobalState({ action: { type: 'NAVIGATE_USER', payload: { userId: '123' } } })`
2. 主应用监听到 `action` 变化，执行路由跳转 `history.push('/user/detail/123')`
3. 用户子应用在 `mount` 生命周期中读取 `props.action`，执行高亮逻辑
4. 操作完成后，主应用将 `action` 重置为 `undefined`，防止重复触发

## 7. 隔离方案

### 7.1 JS 沙箱隔离

使用 qiankun 的 Proxy 沙箱（默认启用），核心机制如下：

- 每个子应用拥有独立的 `window` 代理对象（fakeWindow）
- 子应用对 `window` 的读写操作被代理到 fakeWindow，不影响真实 window
- 子应用卸载时，对 `window` 的所有修改自动回收
- 全局定时器（`setTimeout` / `setInterval`）和事件监听在子应用卸载时自动清除

**注意事项：**

- 子应用不应依赖 `window.__POWERED_BY_QIANKUN__` 以外的全局变量做业务逻辑
- 第三方库（如 ECharts）可能直接操作真实 window，需单独注意

### 7.2 CSS 样式隔离

| 方案 | 隔离强度 | 兼容性 | 弊端 | 适用场景 |
|------|----------|--------|------|----------|
| strictStyleIsolation（Shadow DOM） | 最强 | 一般 | Modal/Tooltip 弹出层可能失效 | 要求严格隔离时 |
| experimentalStyleIsolation（scoped） | 强 | 好 | 动态插入的样式无法覆盖 | **推荐默认使用** |
| CSS Modules + 命名约定 | 中 | 最好 | 需团队规范执行 | 补充方案 |

本项目默认使用 `experimentalStyleIsolation`，若出现 Modal 样式失效，切换为 `strictStyleIsolation` 并配合 `getPopupContainer` 指定挂载节点。

### 7.3 公共依赖共享策略

通过 `webpack externals` 将以下依赖从子应用 bundle 中排除，改为在主应用 HTML 中通过 CDN 引入：

- `react` / `react-dom`（版本 18.x，React 子应用共用）
- `vue`（版本 3.x，Vue 子应用使用，与 React 运行时独立）
- `axios`（所有子应用共用）

> ⚠️ 注意：React 和 Vue 运行时不可共用同一实例，需分别在主应用 HTML 中引入。各子应用通过 externals 引用对应全局变量（`window.React` / `window.Vue`）。

## 8. 性能策略

### 8.1 子应用加载策略

| 策略 | 触发时机 | 配置方式 |
|------|----------|----------|
| 按需加载 | 用户访问对应路由时加载 | 默认行为，无需额外配置 |
| 预加载 | 主应用完成首屏渲染后 | `qiankun prefetchApps()` 配置 |
| 缓存挂载 | 子应用已加载后切换回来时 | `loadMicroApp` 替代 `registerMicroApps` |

### 8.2 预加载优先级

主应用完成首屏渲染（`DOMContentLoaded`）后，按以下优先级顺序预加载子应用：

1. `app-user`（使用频率最高）
2. `app-order`
3. `app-product`、`app-dashboard`

### 8.3 性能基准目标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 主应用首屏 FCP | < 1.5s | Lighthouse / Web Vitals |
| 子应用首次激活耗时 | < 1.0s | performance.mark 打点 |
| 子应用切换耗时（已缓存） | < 200ms | performance.mark 打点 |
| 主应用 JS bundle 大小 | < 200KB（gzip） | webpack-bundle-analyzer |
| 单个子应用 bundle 大小 | < 500KB（gzip） | webpack-bundle-analyzer |

## 9. 部署架构

### 9.1 部署拓扑

各子应用独立构建，产物部署到独立的静态资源路径，主应用通过动态配置中心获取各子应用的入口 URL，不硬编码在代码中。

| 应用 | 构建命令 | 生产入口 URL |
|------|----------|--------------|
| shell-app | `npm run build` | `https://erp.company.com/` |
| app-user | `npm run build` | `https://static.company.com/user/` |
| app-product | `npm run build` | `https://static.company.com/product/` |
| app-order | `npm run build` | `https://static.company.com/order/` |
| app-dashboard | `npm run build` | `https://static.company.com/dashboard/` |

### 9.2 子应用注册配置

主应用在启动时从配置接口拉取子应用注册列表，不在代码中硬编码：

```javascript
// GET /api/micro-apps/config
[
  { name: 'app-user',      entry: '//static.company.com/user/',      activeRule: '/user' },
  { name: 'app-product',   entry: '//static.company.com/product/',   activeRule: '/product' },
  { name: 'app-order',     entry: '//static.company.com/order/',     activeRule: '/order' },
  { name: 'app-dashboard', entry: '//static.company.com/dashboard/', activeRule: '/dashboard' },
]
```

### 9.3 CI/CD 流程

每个仓库独立配置 CI/CD 流水线，推送到 `main` 分支后自动触发构建和部署，与其他子应用的发布互不影响。子应用发布后，主应用无需重新构建即可加载新版本。

## 10. 开发规范

### 10.1 目录结构规范

所有子应用统一使用以下目录结构：

```
app-xxx/
├── src/
│   ├── index.ts          # 微前端入口（export bootstrap/mount/unmount）
│   ├── main.ts           # 独立运行入口
│   ├── pages/            # 页面组件
│   ├── components/       # 通用组件
│   ├── services/         # API 调用层
│   └── stores/           # 应用内状态管理
├── public/
├── webpack.config.js / vite.config.ts
└── package.json
```

### 10.2 子应用必须实现的生命周期

每个子应用的 `src/index.ts` 必须导出以下三个生命周期函数：

```typescript
// 应用初始化时调用一次，适合做全局配置初始化
export async function bootstrap() {}

// 每次激活时调用，负责将应用渲染到指定 DOM 节点
export async function mount(props: any) {}

// 每次失活时调用，负责销毁应用实例、清理副作用
export async function unmount(props: any) {}
```

### 10.3 端口分配规范

| 应用 | 开发端口 | 备注 |
|------|----------|------|
| shell-app | 3000 | 主应用，唯一访问入口 |
| app-user | 3001 | |
| app-product | 3002 | |
| app-order | 3003 | |
| app-dashboard | 3004 | |
| mock-server | 4000 | 本地 Mock API 服务 |

### 10.4 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 子应用名称 | `app-` 前缀 + 模块名（kebab-case） | `app-user`、`app-order` |
| 路由前缀 | 与子应用模块名一致 | `/user`、`/order` |
| 全局状态 action type | 大写 + 下划线 | `NAVIGATE_USER`、`LOGOUT` |
| CSS 类名前缀 | 子应用名缩写 + `-` | `usr-`、`prd-`、`ord-` |
| 环境变量 | `REACT_APP_` 或 `VITE_` + 大写 | `VITE_API_BASE_URL` |
