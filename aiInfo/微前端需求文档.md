## 项目需求：企业级后台管理系统（ERP Lite）

### 项目背景

一个中型企业需要一套内部管理平台，包含多个独立业务模块。这些模块由不同团队维护，技术栈可能不同，需要统一的入口和用户体验。这是微前端的典型应用场景。

---

### 系统架构要求

**主应用（Shell App）** 作为容器，负责：
- 统一的顶部导航栏和侧边菜单
- 用户登录 / 鉴权 / 权限管理
- 全局状态共享（用户信息、主题、语言）
- 子应用的注册、加载与路由调度

**四个子应用**，每个独立开发、独立部署：

| 子应用 | 技术栈要求 | 核心功能 |
|---|---|---|
| 用户管理 App | React 18 | 用户列表、角色权限、组织架构 |
| 商品管理 App | Vue 3 | 商品 CRUD、分类管理、库存预警 |
| 订单管理 App | React 18 | 订单列表、状态流转、数据统计图表 |
| 数据看板 App | 原生 JS 或任意框架 | 核心指标卡片、折线图、地图热力图 |

---

### 核心技术挑战（学习重点）

**第一阶段：基础搭建**
- 用 **qiankun** 或 **single-spa** 搭建主应用框架
- 配置各子应用的入口、激活规则、沙箱隔离
- 解决跨域、webpack publicPath 等工程化问题

**第二阶段：通信机制**
- 主子应用通信：登录后将 `userInfo`、`token` 下发给所有子应用
- 子应用间通信：订单模块点击用户名，跳转到用户管理模块并高亮该用户
- 全局事件总线 vs props 传递，理解两种方式的适用场景

**第三阶段：样式隔离**
- 验证 Shadow DOM 隔离 vs scoped CSS 的区别
- 主题色切换时，主应用的 CSS 变量如何同步到子应用
- 解决 antd / element-plus 全局样式污染问题

**第四阶段：性能优化**
- 子应用预加载策略配置
- 公共依赖（React、Vue）抽取，避免重复加载
- 子应用缓存：切换回已加载的子应用时不重新挂载

**第五阶段：独立部署**
- 每个子应用独立 CI/CD 流水线
- 主应用动态读取子应用注册配置（从接口获取，而非硬编码）
- 灰度发布：同一子应用部署两个版本，按用户分流

### 具体功能页面（用于填充内容，不必全做）

**用户管理子应用**
- 用户列表页：表格 + 搜索 + 分页
- 用户详情页：基本信息 + 操作日志
- 角色管理页：权限树形结构

**商品管理子应用**
- 商品列表：支持多条件筛选
- 商品编辑：富文本描述 + 图片上传
- 库存看板：低库存预警列表

**订单管理子应用**
- 订单列表 + 状态筛选
- 订单详情：时间线展示状态流转
- 销售统计：近30天折线图

**数据看板子应用**
- 4个核心指标卡片（GMV、订单量、用户数、转化率）
- 销售趋势图
- 省份销售热力地图

### 验收标准

完成后你的项目应该能回答这些问题：

1. 子应用是如何做到 JS 隔离的？`window` 对象污染如何防止？
2. 如果商品子应用用了 Vue，订单子应用用了 React，它们能共享同一个全局 Store 吗？怎么做？
3. 子应用 A 修改了 `document.title`，切换到子应用 B 后如何还原？
4. 如何在不刷新页面的情况下，将子应用从 v1 热更新到 v2？
5. 主应用崩溃和子应用崩溃的边界如何隔离？

---

### 建议学习顺序

```
搭主应用骨架
  → 接入第一个 React 子应用（用户管理）
    → 接入第一个 Vue 子应用（商品管理）
      → 打通主子通信
        → 解决样式隔离问题
          → 接入剩余子应用
            → 性能优化
              → 模拟独立部署
```
