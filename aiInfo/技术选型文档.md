## 1. 选型原则

在技术选型时，所有决策遵循以下优先级顺序：

1. **稳定性优先**：优先选择社区成熟、长期维护的方案，避免引入实验性库
2. **团队熟悉度**：在同等条件下，优先选择团队已有实践经验的技术
3. **生态兼容性**：所选技术需与微前端架构（qiankun）良好兼容，无明显冲突
4. **最小化引入**：不因为"可能用到"而引入依赖，每一个依赖都需要有明确的使用场景
5. **版本锁定**：主应用与各子应用的共享依赖必须锁定到相同的大版本，防止运行时冲突

## 2. 微前端框架

### 2.1 候选方案对比

| 维度 | qiankun | single-spa | Module Federation | iframe |
|------|---------|------------|-------------------|--------|
| JS 沙箱 | ✅ Proxy 沙箱（内置） | ❌ 需自行实现 | ⚠️ 无隔离 | ✅ 天然隔离 |
| CSS 隔离 | ✅ Shadow DOM / scoped（内置） | ❌ 需自行实现 | ❌ 无隔离 | ✅ 天然隔离 |
| 接入成本 | 低（改造量小） | 高（需大量配置） | 中（webpack 配置） | 低 |
| 子应用通信 | ✅ initGlobalState + props | ✅ 自定义事件 | ✅ 共享模块 | ❌ postMessage，繁琐 |
| 独立部署 | ✅ | ✅ | ✅ | ✅ |
| 技术栈无关 | ✅ | ✅ | ⚠️ 需统一 webpack | ✅ |
| 社区活跃度 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | — |
| 国内案例 | 丰富（阿里系） | 较少 | 较少 | — |

### 2.2 决策结果

**选用：qiankun v2.x**

### 2.3 选用理由

- 内置 Proxy 沙箱和 CSS 隔离，无需额外实现核心能力，大幅降低维护成本
- 基于 single-spa 封装，底层可靠，上层 API 更友好
- 国内社区资料丰富，团队遇到问题时容易找到解决方案
- 阿里内部大规模验证，稳定性有保障

### 2.4 已知风险与应对

| 风险 | 描述 | 应对方案 |
|------|------|----------|
| Shadow DOM 兼容问题 | strictStyleIsolation 模式下，Modal/Tooltip 弹层可能丢失样式 | 默认使用 experimentalStyleIsolation，问题应用单独处理 |
| Proxy 沙箱兼容性 | 不支持 IE11（Proxy 无法 polyfill） | 本项目明确不支持 IE，不受影响 |
| qiankun 版本停更风险 | qiankun 3.x 开发中，2.x 可能逐渐停止维护 | 关注官方动态，3.x 稳定后评估升级 |

## 3. 构建工具

### 3.1 候选方案对比

| 维度 | Webpack 5 | Vite 5 | Rspack |
|------|-----------|--------|--------|
| 构建速度（开发） | 慢 | ⭐ 极快（ESM + esbuild） | ⭐ 快（Rust） |
| 构建速度（生产） | 中 | 中（Rollup） | 快 |
| qiankun 兼容性 | ✅ 官方支持，文档完善 | ⚠️ 需额外配置 vite-plugin-qiankun | ⚠️ 社区方案，不够成熟 |
| 生态丰富度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 配置复杂度 | 高 | 低 | 中 |
| 团队熟悉度 | 高 | 中 | 低 |

### 3.2 决策结果

- **主应用 & React 子应用**：Webpack 5（与 qiankun 官方示例保持一致，降低踩坑风险）
- **Vue 子应用 & Dashboard**：Vite 5（Vue 官方推荐，开发体验好，通过 vite-plugin-qiankun 接入）

### 3.3 选用理由

主应用使用 Webpack 5 的核心原因是 qiankun 的官方文档和示例均基于 webpack，遇到问题时有充分的社区参考。Vue 子应用使用 Vite 是因为 Vue 3 的官方脚手架默认 Vite，团队迁移成本低，且 vite-plugin-qiankun 已经过社区验证。

### 3.4 关键配置要求

无论使用哪种构建工具，所有子应用必须满足以下构建要求：

```javascript
// 必须配置 output.library，让主应用可以获取生命周期函数
output: {
  library: `${packageName}-[name]`,
  libraryTarget: 'umd',
  chunkLoadingGlobal: `webpackJsonp_${packageName}`,
}

// 必须配置 publicPath，否则子应用资源路径会出错
output: {
  publicPath: process.env.NODE_ENV === 'production'
    ? 'https://static.company.com/app-xxx/'
    : 'http://localhost:300x/',
}
```

## 4. 前端框架

### 4.1 决策结果

| 应用 | 框架 | 版本 | 选用理由 |
|------|------|------|----------|
| shell-app | React | 18.x | 团队主力技术栈，Concurrent Mode 性能更优 |
| app-user | React | 18.x | 同上 |
| app-order | React | 18.x | 同上 |
| app-product | Vue | 3.x | 演示微前端多技术栈共存场景 |
| app-dashboard | 原生 JS | — | 演示无框架子应用接入，适合纯展示型应用 |

### 4.2 版本统一要求

**React 子应用必须使用完全相同的 React 版本（18.x）**，且通过 externals 共享同一个 React 实例，避免出现多个 React 实例导致的 Hook 报错：

```
Error: Invalid hook call. Hooks can only be called inside of the body of a function component.
```

Vue 子应用同理，共享同一个 Vue 实例。React 与 Vue 的运行时相互独立，不存在冲突。

## 5. 路由管理

### 5.1 决策结果

| 应用 | 路由库 | 版本 | 模式 |
|------|--------|------|------|
| shell-app | React Router | 6.x | History 模式 |
| app-user | React Router | 6.x | History 模式 |
| app-order | React Router | 6.x | History 模式 |
| app-product | Vue Router | 4.x | History 模式 |
| app-dashboard | 无（原生 JS 处理） | — | — |

### 5.2 重要配置：路由 base

子应用内部路由必须设置 `base`，否则子应用路由会覆盖主应用路由：

```typescript
// React 子应用（app-user）
<BrowserRouter basename="/user">
  ...
</BrowserRouter>

// Vue 子应用（app-product）
createRouter({
  history: createWebHistory('/product'),
  routes: [...]
})
```

### 5.3 不使用 Hash 模式的理由

Hash 模式（`#/user/list`）会导致多个子应用的路由难以区分，且 SEO 不友好。本项目统一使用 History 模式，由 Nginx 配置 `try_files` 处理 404 回退。

## 6. 状态管理

### 6.1 候选方案对比（React 应用）

| 方案 | 体积 | 学习曲线 | 适合场景 | 与 React 18 兼容性 |
|------|------|----------|----------|-------------------|
| Zustand | 极小（~1KB） | 低 | 中小型应用，轻量状态 | ✅ 完美 |
| Redux Toolkit | 中（~15KB） | 中 | 大型应用，复杂状态流转 | ✅ |
| Jotai | 小（~3KB） | 低 | 原子化状态，细粒度更新 | ✅ |
| Context API | 0（内置） | 低 | 简单的全局数据共享 | ✅ |

### 6.2 决策结果

- **React 子应用**：Zustand v4.x
- **Vue 子应用**：Pinia v2.x

### 6.3 选用理由

子应用的状态管理只需处理**应用内部**的业务状态，不需要跨应用共享，因此优先选择轻量方案。Zustand API 简洁，无需 Provider 包裹，与 qiankun 的 mount/unmount 生命周期配合更简单。Pinia 是 Vue 3 官方推荐，取代 Vuex 4。

### 6.4 跨应用状态说明

跨应用的全局状态（用户信息、主题、权限）**不使用** Zustand/Pinia，而是通过 qiankun 的 `initGlobalState` 管理，详见架构设计文档第 6 章。

## 7. UI 组件库

### 7.1 决策结果

| 应用 | 组件库 | 版本 | 说明 |
|------|--------|------|------|
| shell-app | Ant Design | 5.x | 主应用导航、布局组件 |
| app-user | Ant Design | 5.x | 与主应用统一视觉风格 |
| app-order | Ant Design | 5.x | 与主应用统一视觉风格 |
| app-product | Element Plus | 2.x | Vue 生态主流选择 |
| app-dashboard | ECharts | 5.x | 纯图表展示，无需通用组件库 |

### 7.2 样式隔离注意事项

Ant Design 和 Element Plus 都会向全局注入 CSS 样式，在微前端场景下存在样式污染风险：

**Ant Design（子应用）配置方案：**

```typescript
// 使用 ConfigProvider 的 prefixCls 隔离类名前缀
<ConfigProvider prefixCls="user-ant">
  <App />
</ConfigProvider>
```

**Element Plus（子应用）配置方案：**

```typescript
// 使用 namespace 隔离类名前缀
app.use(ElementPlus, { namespace: 'prd-ep' })
```

### 7.3 主题统一方案

主应用通过 CSS 变量向子应用注入主题色，各子应用的组件库主题色读取相同的 CSS 变量，保证视觉统一：

```css
/* 主应用定义 */
:root {
  --primary-color: #1677ff;
  --success-color: #52c41a;
  --warning-color: #faad14;
  --error-color:   #ff4d4f;
}
```

## 8. HTTP 客户端

### 8.1 决策结果

**全部应用统一使用：Axios v1.x**

Axios 通过 `externals` 在所有应用间共享同一实例，避免重复打包。

### 8.2 封装规范

每个子应用基于 Axios 封装自己的请求实例，**不共享** Axios 实例对象，只共享 Axios 库本身：

```typescript
// 每个子应用 src/services/request.ts
import axios from 'axios'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000,
})

// 请求拦截：注入 token
request.interceptors.request.use(config => {
  const token = window.__MICRO_APP_TOKEN__ // 由主应用通过 props 注入
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// 响应拦截：统一错误处理
request.interceptors.response.use(
  res => res.data,
  err => {
    if (err.response?.status === 401) {
      // 通知主应用登出
    }
    return Promise.reject(err)
  }
)

export default request
```

## 9. CSS 方案

### 9.1 决策结果

| 应用 | CSS 方案 | 说明 |
|------|----------|------|
| shell-app | CSS Modules + CSS 变量 | 主应用布局样式，定义全局 CSS 变量 |
| app-user | CSS Modules | 防止类名冲突 |
| app-order | CSS Modules | 防止类名冲突 |
| app-product | `<style scoped>` | Vue 单文件组件内置方案 |
| app-dashboard | BEM 命名约定 | 原生 JS 应用，手动管理类名 |

### 9.2 全局样式规范

- 子应用**禁止**修改 `:root` 下的 CSS 变量（只能读取）
- 子应用**禁止**设置 `body`、`html` 的全局样式
- 子应用所有样式选择器必须以子应用专属前缀开头（如 `.usr-`、`.ord-`）

### 10.1 代码规范

| 工具 | 版本 | 用途 |

| ESLint | 8.x | JS/TS 代码检查 |
| Prettier | 3.x | 代码格式化 |

| lint-staged | 14.x | 提交前只检查暂存文件 |
| commitlint | 17.x | Git commit 信息规范检查 |

### 10.2 TypeScript

全部应用启用 TypeScript，统一使用以下 `tsconfig.json` 基础配置：

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "strict": true,
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  }
}
```

### 10.3 包管理器

**统一使用 pnpm v8.x**，原因：

- monorepo 场景下对磁盘空间和 node_modules 管理更优
- 严格的依赖解析，防止幽灵依赖（phantom dependencies）
- workspace 功能方便管理主应用和子应用的公共脚本

### 10.4 Monorepo 结构

```
erp-lite/
├── packages/
│   ├── shell-app/
│   ├── app-user/
│   ├── app-product/
│   ├── app-order/
│   └── app-dashboard/
├── shared/
│   ├── types/            # 跨应用共享的 TypeScript 类型定义
│   └── utils/            # 跨应用共享的纯函数工具（不含框架依赖）
├── pnpm-workspace.yaml
└── package.json
```

> ⚠️ `shared/` 中只允许放置**纯函数工具和类型定义**，禁止引入任何框架（React/Vue）相关代码，保证各子应用可以独立使用。

## 11. 监控与质量

### 11.1 当前阶段决策

本期暂不引入线上监控 SDK，由以下工具保障开发质量：

| 工具 | 用途 |

| Vue Test Utils | Vue 组件测试 |
| Playwright | E2E 集成测试 |
| webpack-bundle-analyzer | bundle 体积分析 |

### 11.2 后续阶段计划

第四阶段完成性能优化后，评估引入以下方案：

- **前端监控**：Sentry（错误监控）
- **性能采集**：Web Vitals SDK + 自建上报接口
- **子应用加载耗时**：`performance.mark` 埋点，上报到 APM
